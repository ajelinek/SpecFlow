## What this workflow is for

`201-high-level-design` creates a short feature overview that explains what the feature does, what is in scope, what is out of scope, how the user journey works, and, when useful, how reviewers will know the feature is working correctly.

## When to use it

- When you are starting design work for a specific feature
- When the team needs a shared plain-language understanding before writing specs
- When you want a reviewable scope boundary before implementation begins

## What it produces

- `.specflow/features/<feature-name>/overview.md`
- Front matter with `feature`, optional `fid`, and `status`
- A concise feature description
- In-scope and out-of-scope boundaries
- A plain-text primary user journey and, when meaningful, acceptance criteria

## Required inputs

- A feature name
- One sentence of context about what the feature should do

If `D10-feature-overview.md` already tracks this feature, `201` may reuse its `F-ID` or ask whether to add the feature there first.

## What usually comes next

Most of the time, `202-spec-design` comes next so the expected behavior is written down before implementation starts.

## Practical guidance

- Keep the output short enough for fast human review.
- Use the document to draw scope boundaries, not to explain implementation details.
- Make the out-of-scope list explicit so later work does not quietly expand.
- The user journey should stay a plain-text bullet list, not a diagram.
- The written file starts with lifecycle front matter so later workflows can track status in place.

## Common mistakes

- Treating `201` as an implementation plan
- Writing generic scope bullets that do not actually constrain the feature
- Skipping explicit out-of-scope decisions

<Callout tone="core" title="Why it matters">
  `201` is the moment where SpecFlow turns a vague request into something reviewable. If the feature definition is blurry here, everything after it gets harder.
</Callout>
