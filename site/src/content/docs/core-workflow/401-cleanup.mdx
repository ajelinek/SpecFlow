## What this workflow is for

`401-cleanup` performs a deliberate cleanup pass on an already-changed scope. It is designed to reduce duplication, improve naming and structure, and fix obvious in-scope robustness issues without turning the work into a broad rewrite. For test cleanup, it is still cleanup-only work: no net-new coverage.

## When to use it

- After `301-spec-implementation`
- When a change already exists and now needs a cleanup pass
- When you want cleanup to stay reviewable and tightly scoped

## What it produces

- Cleaner source files or cleaner test files
- Validation output proving the cleanup stayed safe
- A summary of any out-of-scope opportunities that were not approved

## Required inputs

- A cleanup scope anchor such as staged changes, a file set, or a diff range
- Exactly one cleanup target type: source code or test code

If the selected changed set includes both source and test files, `401` stops and asks you to choose one side before it can proceed.

## Common prompts

<PromptBlock title="Prompt" text={`Run 401-cleanup on the changed source files.`} />

<PromptBlock title="Prompt" text={`Run 401-cleanup on the changed test files.`} />

## What usually comes next

If you cleaned source code first and still want test cleanup, run `401-cleanup` again as a separate pass with the test target type.

## Practical guidance

- Treat the user-defined changed set as the default boundary.
- Ask for explicit approval before expanding scope beyond that boundary.
- Fix obvious in-scope risks when intent is clear, but do not disguise feature work as cleanup.
- Test cleanup is cleanup-only. It must not add new coverage, new scenarios, or new test files.

## Common mistakes

- Trying to clean up source and test files in one pass
- Pointing `401` at a mixed changed set without choosing source or test first
- Using `401` for new feature delivery instead of post-change cleanup
- Treating out-of-scope duplication as automatic permission to refactor broadly

<Callout tone="core" title="One target type per run">
  This workflow is intentionally strict: source cleanup and test cleanup are separate operations. That constraint keeps cleanup smaller, clearer, and easier to review.
</Callout>
