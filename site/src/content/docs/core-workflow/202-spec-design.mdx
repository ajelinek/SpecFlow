## What this workflow is for

`202-spec-design` writes Gherkin scenarios for a feature. It captures what the system should do in user-visible terms, supports multiple coverage levels, and continues existing numbering when a spec file already exists.

## When to use it

- After `201-high-level-design`
- When you want behavior to be reviewable before code is written
- When you need a shared testable description of the feature

## What it produces

- `.specflow/features/<feature-name>/specs.feature`
- Numbered `TS#` scenarios grouped into `TSM#` test modules
- Lifecycle tags such as `@status_pending` plus path tags such as `@happyPath`, `@errorPath`, and `@edgePath`
- Updates to existing scenarios in place when that is the clearest way to extend coverage

## Required inputs

- Feature name
- Coverage level: Happy Path Only, Balanced, or Comprehensive

## What usually comes next

`203-implementation-design` is the optional next step when you want a persisted, repo-grounded implementation plan. `204-feature-validation` can then review `overview.md`, `specs.feature`, and `implementation.md` together before coding. Otherwise, teams often go straight to `301-spec-implementation`.

## Practical guidance

- Start with Happy Path Only if you want a fast first pass.
- Use Balanced when the feature has meaningful error states you do not want to forget.
- Keep steps user-observable. Avoid implementation detail in the scenario text.
- Extend an existing scenario when the extra coverage belongs to the same business outcome instead of creating a near-duplicate scenario.

## Common mistakes

- Treating scenarios like unit tests instead of business outcomes
- Writing every tiny UI action as a separate scenario
- Asking `202` to invent a new feature definition that should have been resolved in `201`
- Treating `202` as append-only when an existing scenario should just be extended

<Callout tone="tip" title="Coverage discipline">
  The goal is not maximum scenario count. The goal is enough behavior coverage to guide implementation and review without creating noise.
</Callout>
