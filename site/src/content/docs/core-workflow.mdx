The core delivery loop is `201 -> 202 -> 301 -> 401`. It lives inside the larger lifecycle where the `100` series builds project context and `110` maps the feature backlog.

## The larger groupings

| Grouping | What it means |
|---|---|
| `100` series | Build shared project understanding: domain, product, architecture, UX, and feature map |
| `200` series | Define one feature before coding |
| `300` series | Implement or repair behavior |
| `400` series | Run cleanup-only refinement on an already-changed scope |

<Callout tone="core" title="Use the 100-series earlier than you think">
  You get better results when you use the `100` series on both new and existing projects that still have weak domain, product, architecture, or UX context.
</Callout>

## Lean delivery loop

| Step | Skill | What it does |
|---|---|---|
| 1 | `201-high-level-design` | Plain-language feature overview with scope and user journey |
| 2 | `202-spec-design` | Gherkin scenarios describing expected behavior |
| 3 | `301-spec-implementation` | Tests-first implementation with internal cleanup and validation |
| 4 | `401-cleanup` | Separate cleanup-only pass for a changed source or test scope |

<Callout tone="tip" title="Where this loop fits">
  Use this loop when the project context already exists, or after the `100` series and `110` have created that context for you.
</Callout>

- **[201 High-Level Design](./201-high-level-design/)** — define the feature
- **[202 Spec Design](./202-spec-design/)** — write the behavior
- **[301 Spec Implementation](./301-spec-implementation/)** — build against the spec
- **[401 Cleanup](./401-cleanup/)** — optional separate cleanup pass
