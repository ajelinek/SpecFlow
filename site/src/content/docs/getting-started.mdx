SpecFlow is a lightweight, spec-driven workflow for building software with AI. It provides a structured catalog of skills (100-402 series) and optional worker agents that guide you through project setup, feature design, implementation, correction, and cleanup without replacing your coding standards or framework rules.

## 1. Install the skills

Install interactively and choose only the skills you want:

```bash
npx skills add ajelinek/SpecFlow
```

Install the full catalog:

```bash
npx skills add ajelinek/SpecFlow --skill '*'
```

Use `npx skills add ajelinek/SpecFlow --list` if you want to preview the available skills first.

<Callout tone="tip" title="Skills install versus agents install">
  `npx skills add ...` installs the SpecFlow skills. The bundled worker agents are a separate optional install.
</Callout>

## 2. Install the optional worker agents

If you want the supporting `coder`, `execution-agent`, and `designer` workers, use **[Custom Agents](./custom-agents/)** next.

<Callout tone="tip" title="When the agents matter most">
  The bundled workers are the intended support structure for full `301`, `302`, `401`, and `402` runs. You can start without them, but stronger orchestration comes from installing them.
</Callout>

## 3. Run the lean default

If the repo already has enough context, the lean path is the fastest clean start. You can use the skill numbers as a shorthand in your prompts (e.g., saying "Run 301" instead of the full name):

<PromptBlock
  title="Run this"
  text={`Run 201 for invoice approval.
Use 202 for invoice approval with balanced coverage.
Run 301 for the invoice approval feature.
Run 401 on the changed source files.`}
/>

Run `401-cleanup` again later for changed test files if you want a separate test cleanup pass too.

## 4. Pick the right path when the lean flow is not enough

- Read **[Project Foundation](../project-foundation/)** for the larger lifecycle, groupings, and why the `100` series helps on both new and existing projects.
- Use the homepage flow visualization when you want to compare the full, conservative, lean, and implementation-first paths side by side.

<Callout tone="tip" title="What 301 does when artifacts are missing">
  `301` can synthesize missing `201`, `202`, and `203` outputs in memory and run `204-feature-validation` before coding. Run `203-implementation-design` yourself first when you want a persisted `.specflow/features/<fid>-<feature-slug>/implementation.md` file.
</Callout>

<Callout tone="tip" title="What the worker agents are for">
  Full `301`, `302`, `401`, and `402` runs are designed around separate workers: `coder` for scoped code changes, `execution-agent` for test/lint/build validation, and `designer` for UI-affecting design direction.
</Callout>
