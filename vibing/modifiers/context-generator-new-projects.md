# New Project Context Generator

**Purpose**: Generates all AGENT.md context files for greenfield projects by reading design documents and analyzing the complete vibing rules and agents directory.

## Overview

This modifier reads design documents from `_docs/design/` directory and generates comprehensive AGENT.md context files following the standardized AGENT.md specification. It analyzes all rules and agents to assign appropriate responsibilities and rule references.

## Inputs

- **Project Root Path**: The root directory of the project
- **Design Documents**: `_docs/design/` directory with architecture documents
- **Technology Stack Preferences**: Optional preferences for technology choices
- **Rules Directory**: Complete analysis of `vibing/rules/` directory
- **Agents Directory**: Complete analysis of `vibing/agents/` directory

## Required Design Documents

The generator expects these design documents in `_docs/design/`:

- `Project_Overview.md` - Business context and project purpose
- `System_Architecture.md` - Overall architecture and technology stack
- `Data_Model.md` - Entity definitions and relationships
- `Backend_Architecture.md` - API and service layer patterns
- `Frontend_Architecture.md` - UI framework and component architecture
- `09 - Data Access Patterns and Objects.md` - Data layer patterns

## Agent-to-Component Mapping

The generator maps components to responsible agents using this matrix:

| Component/Directory     | Responsible Agent                          | Applicable Rules                            |
| ----------------------- | ------------------------------------------ | ------------------------------------------- |
| `api/` or `backend/`    | @vibing/agents/backend-engineer.md         | backend, foundation, data, testing          |
| `ui/` or `frontend/`    | @vibing/agents/frontend-engineer.md        | framework-specific, ui, foundation, testing |
| `ui/store/` or `store/` | @vibing/agents/data-engineer.md            | apollo/data-management, foundation          |
| `e2e/`                  | @vibing/agents/test-automation-engineer.md | testing/e2e                                 |
| `e2e/pages/`            | @vibing/agents/test-automation-engineer.md | testing/page-objects                        |
| `data/`                 | @vibing/agents/data-engineer.md            | data/persistence, foundation                |
| Components (foundation) | @vibing/agents/frontend-engineer.md        | ui/foundational-components                  |
| Components (features)   | @vibing/agents/frontend-engineer.md        | ui/component-guidelines                     |

## Generated Files

### 1. Global Context (Pre-created)

**Note**: The global `vibing/context/AGENT.md` file should be pre-created and will not be generated by this modifier.
**Note**: No new rules should be created and all referenced rules and agents must already exist.

### 2. Backend Context (`{backend-dir}/AGENT.md`)

**Purpose**: Backend/API specific context with service layer patterns

**Content**:

- Backend overview and purpose (from Backend_Architecture.md)
- Backend technology stack
- API design patterns with agent assignments
- Service layer structure with responsibilities
- Data access patterns with agent assignments
- Authentication approach
- Error handling conventions
- Backend-specific rule references
- **Responsible Agent**: @vibing/agents/backend-engineer.md

### 3. Frontend Context (`{frontend-dir}/AGENT.md`)

**Purpose**: Frontend/UI specific context with component architecture

**Content**:

- Frontend overview and purpose (from Frontend_Architecture.md)
- Frontend technology stack
- Component architecture with agent assignments
- Directory structure with responsibilities
- State management strategy
- Styling approach
- Testing patterns
- Framework-specific rule references
- **Responsible Agent**: @vibing/agents/frontend-engineer.md

### 4. Store Context (`{frontend-dir}/store/AGENT.md`)

- Data management rule references
- **Responsible Agent**: @vibing/agents/data-engineer.md

### 5. E2E Context (`{e2e-dir}/AGENT.md`)

- E2E testing overview
- Testing framework setup

- **Responsible Agent**: @vibing/agents/test-automation-engineer.md

### 6. Page Objects Context (`{e2e-dir}/pages/AGENT.md`)

- Page objects overview
- Page object structure with responsibilities

- **Responsible Agent**: @vibing/agents/test-automation-engineer.md

### 7. Data Context (`{data-dir}/AGENT.md`)

- Data layer overview
- Database technology

- Schema overview
- Migration patterns

- **Responsible Agent**: @vibing/agents/data-engineer.md

## Process

### Phase 1: Validation and Discovery

1. **Validate Design Documents**:

   - Check for required design documents in `_docs/design/`
   - Verify document completeness and structure
   - Report missing or incomplete documents

2. **Parse Design Documents**:

   - Extract project overview from `Project_Overview.md`
   - Extract technology stack from `System_Architecture.md`
   - Extract data model from `Data_Model.md`
   - Extract backend patterns from `Backend_Architecture.md`
   - Extract frontend patterns from `Frontend_Architecture.md`
   - Extract data access patterns from `09 - Data Access Patterns and Objects.md`

3. **Analyze Rules and Agents**:

   - Scan all files in `vibing/rules/` directory recursively
   - Scan all files in `vibing/agents/` directory recursively
   - Categorize rules by technology and component type
   - Map agents to component responsibilities

4. **Detect Technology Stack**:

   - Parse System_Architecture.md for technology choices
   - Identify frontend framework (React, SolidJS, Astro)
   - Identify backend framework (Express, Fastify, etc.)
   - Identify data layer (Firebase, PostgreSQL, etc.)
   - Identify testing framework (Playwright, Vitest, Jest)
   - Identify build tools (Vite, Webpack)

5. **Determine Directory Structure**:
   - Infer directory structure from design documents
   - Map component types to directory patterns
   - Assign agent responsibilities based on component types

### Phase 2: Rule and Agent Mapping

1. **Technology-to-Rule Mapping**:

   - **Always Include**: Foundation rules, testing rules, error handling
   - **Frontend Framework Rules**: React, SolidJS, or Astro specific rules
   - **UI/UX Rules**: All common UI rules for frontend projects
   - **GraphQL/Apollo Rules**: If Apollo Client detected
   - **Backend Rules**: All backend-specific rules
   - **Data Rules**: All data persistence rules

2. **Component-to-Agent Mapping**:
   - Map backend components to backend-engineer.md
   - Map frontend components to frontend-engineer.md
   - Map data store components to data-engineer.md
   - Map E2E components to test-automation-engineer.md
   - Map data layer components to data-engineer.md

### Phase 3: Context Generation

1. **Generate Backend Context** (`{backend-dir}/AGENT.md`):

   - Backend overview from Backend_Architecture.md
   - API design patterns with agent assignments
   - Service layer structure with responsibilities
   - Data access patterns with agent assignments
   - Authentication approach
   - Error handling conventions
   - Backend-specific rule references
   - Responsible agent assignment

2. **Generate Frontend Context** (`{frontend-dir}/AGENT.md`):

   - Frontend overview from Frontend_Architecture.md
   - Component architecture with agent assignments
   - Directory structure with responsibilities
   - State management strategy
   - Styling approach
   - Testing patterns
   - Framework-specific rule references
   - Responsible agent assignment

3. **Generate Store Context** (`{frontend-dir}/store/AGENT.md`):

   - Store overview from Frontend_Architecture.md + Data Access Patterns
   - Service-Repository pattern with responsibilities
   - GraphQL/REST client setup
   - Data fetching conventions
   - Hook patterns
   - Data management rule references
   - Responsible agent assignment

4. **Generate E2E Context** (`{e2e-dir}/AGENT.md`):

   - E2E overview from System_Architecture.md testing section
   - E2E framework setup
   - Page object patterns
   - Selector conventions
   - Test data management
   - Testing rule references
   - Responsible agent assignment

5. **Generate Page Objects Context** (`{e2e-dir}/pages/AGENT.md`):

   - Page objects overview
   - Page object structure with responsibilities
   - Selector naming conventions
   - Interaction patterns
   - Page object pattern rule references
   - Responsible agent assignment

6. **Generate Data Context** (`{data-dir}/AGENT.md`):
   - Data layer overview from Data_Model.md
   - Entity definitions with responsibilities
   - Schema overview
   - Migration patterns
   - Test data utilities
   - Data persistence rule references
   - Responsible agent assignment

### Phase 4: Validation and Reporting

1. **Validate Generated Files**:

   - Verify all generated files are valid markdown
   - Check that all rule references exist in vibing/rules/
   - Check that all agent references exist in vibing/agents/
   - Validate cross-references between context files

2. **Generate Summary Report**:
   - List all generated files
   - Report any missing design documents
   - Report any skipped context files
   - Provide guidance on manual review needs

## Error Handling

- **Missing Design Documents**: Warn about missing documents and suggest alternatives
- **Incomplete Documents**: Report incomplete sections and suggest completion
- **Missing Rules**: Warn if referenced rules don't exist
- **Missing Agents**: Warn if referenced agents don't exist
- **Partial Generation**: Support generating available context files even if some are missing

## Integration

This modifier works alongside:

1. **`technology-stack-analyzer.md`**: Use for tech stack detection validation
2. **`test-rules-customization.md`**: Apply entity customization after generation
3. **`agent-path-updater.md`**: Ensure correct rule and agent references

## Next Steps

After generation:

1. Review generated AGENT.md files for accuracy
2. Verify agent responsibility assignments
3. Validate rule references
4. Run `technology-stack-analyzer.md` to validate filtering
5. Run `test-rules-customization.md` to customize test entities
6. Manually enhance with project-specific details
7. Commit to version control
8. Update as architecture evolves
